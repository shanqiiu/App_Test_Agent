# mainFlow 诊断优化报告

## 基本信息

| 项目 | 内容 |
|---|---|
| 诊断文件 | `shopping-flow-search-and-buy_case_nothink.json` |
| 原始步骤数 | 18 |
| 改进后步骤数 | 15 |
| 诊断日期 | 2026-05-27 |
| 场景 | shopping — 搜索商品并完成购买 |

---

## 一、评估方法

从**衔接性、逻辑清晰性、内容重复性、顺序合理性、数据一致性**五个维度进行系统评估。评估视角聚焦于：当 AI Agent 以 mainFlow 为输入生成仿真 APP 时，各维度的问题会产生何种负面影响。

---

## 二、各维度诊断结果

### 2.1 步骤衔接性 — 等级：严重

| 跳转 | 问题描述 |
|---|---|
| Step 1→2 | 同一首页重复描述，Step1 搜索框有"海尔洗衣机"，Step2 消失，中间无用户动作 |
| Step 2→3 | 搜索框突然从"海尔洗衣机"变为"华为手机"，缺少输入/清空动作 |
| Step 6→7 | 搜索结果页直接跳到"热卖榜"，缺少点击进入的过渡 |
| Step 9→10→11→12 | 四步均在搜索结果页附近循环，无明确页面推进 |
| Step 12→13 | 搜索页突然回到首页，无导航动作 |
| Step 13→14 | 首页跳到"消息界面"，完全偏离购买主线 |
| Step 14→15 | 消息页跳到购物车，无切换动作 |

**根因**：每步描述的是"页面状态快照"，而非"用户动作 → 系统响应"的因果链。AI Agent 需逆向推测页面间的事件，极易生成错误的跳转逻辑。

### 2.2 逻辑清晰性 — 等级：致命

- **描述与数据矛盾**：`description` 写"搜索**iPhone** 手机"，18 个步骤全部描述"**华为**手机"（畅享70X、nova15），AI Agent 在 prompt 中收到冲突信息
- **价格不一致**：同一商品华为畅享70X 256GB — Step7 ¥1400.8、Step16 ¥1648、Step17 补贴价¥1648/国补后¥1400.8。Agent 不知该绑定哪个数据
- **无关步骤**：Step14 消息界面与"搜索→购买"完全无关，Agent 会为不需要的页面生成 UI 和路由

### 2.3 内容重复性 — 等级：严重

按页面聚类后，18 步中实际只有约 8 个有效状态：

| 页面 | 涉及 Step | 实质差异 |
|---|---|---|
| 首页 | 1, 2, 13 | Step1/2 几乎相同 |
| 搜索/筛选 | 3, 4, 5, 10, 11, 12 | Step10/11/12 几乎相同 |
| 热卖榜 | 7, 8, 9 | 同一页面三个描述 |
| 订单确认 | 16, 17 | 同一页面两个描述 |

→ **~10 步是冗余的**，浪费 AI Agent 的 context window，稀释关键信息密度。

### 2.4 顺序合理性 — 等级：严重

```
标准搜索购买流程:
Home → Search → SearchResult → ProductDetail → Cart → Checkout → Payment → OrderDetail

该 mainFlow 实际路径:
Home(×2) → Search → Filter → SearchResult → HotList(×3) → SearchResult(×3) → Home → Messages(无关) → Cart → Checkout(×2) → Payment
```

**缺失关键页面**：ProductDetail（商品详情大图+规格选择+加购按钮）、OrderDetail（支付成功结果页）。

**多余页面**：Messages 消息界面。

### 2.5 数据一致性 — 等级：中等

| 数据项 | 不一致表现 |
|---|---|
| 搜索商品 | description: iPhone / steps: 华为手机 |
| 商品价格 | Step7 ¥1400.8 / Step16 ¥1648 / Step17 双价并存 |
| 购物车角标 | Step13 显示 54 件 / 场景只买 1 件 |

## 三、对 AI Agent 生成的影响汇总

| 危害等级 | 问题维度 | 对 Agent 的具体影响 |
|---|---|---|
| **致命** | 描述(iPhone) vs 数据(华为) 矛盾 | Agent 在 prompt 中收到冲突信息，可能随机选择一端 |
| **致命** | 页面状态式描述而非动作驱动 | Agent 需逆向推测事件，极易生成错误的页面跳转逻辑 |
| **严重** | 18步中 ~10步冗余/循环 | 消耗 context window，Agent 可能对同一页面反复生成 |
| **严重** | Step14 消息页面(无关步骤) | Agent 会为不需要的页面生成 UI 和路由 |
| **中等** | 缺失 ProductDetail 过渡 | Agent 可能从搜索结果直接跳购物车，漏掉规格选择/加购按钮 |
| **中等** | 价格数据不一致 | 数据绑定无单一真相源，Agent 可能随机取值 |

---

## 四、改进原则

1. **动作驱动**：每步 = 用户做了什么 → 页面跳转到哪 → 页面展示了什么关键信息
2. **线性单向**：不走回头路，每步推进到一个新页面状态
3. **一步一页**：每个 step 对应唯一 `targetPage`，不做多步描述同一页
4. **数据一致**：商品名称、价格、SKU 从头到尾不变
5. **对齐描述**：flow.name、description、steps 中的商品/品牌保持一致
6. **去噪精简**：与主线无关的页面/步骤直接删除
7. **增加页面标识**：每步新增 `targetPage` 字段，直接对应 `screenKeys` 中的页面

## 五、改进前后对照

| 维度 | 原始 | 改进后 |
|---|---|---|
| 步骤数 | 18（~10 步冗余/循环） | 15（每步独立推进） |
| 描述范式 | 页面状态快照 | 动作驱动（用户动作→页面→关键信息） |
| 商品一致性 | iPhone描述 vs 华为数据 | 全程 iPhone 16 Pro 256GB 沙漠钛金属 |
| 价格一致性 | 1400 ↔ 1648 横跳 | 全程 ¥6999 → ¥6799（优惠后） |
| 页面标识 | 无 | 每步包含 `targetPage` |
| 无关页面 | 含消息界面 | 已删除 |
| 缺失页面 | 无ProductDetail/OrderDetail | 已补充完整 |
