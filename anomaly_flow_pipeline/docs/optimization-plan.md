# anomaly_flow_pipeline 优化计划

> 基于 phase2_flow.json 五维度评估 + 步骤去冗问题的系统优化方案

---

## 背景

### 输入

对 `outputs/demo/phase2_flow.json` 的 mainFlow（18 步骤的购物流程）从五个维度进行系统评估，每个维度聚焦：**当 AI Agent 以此 mainFlow 为输入生成仿真 APP 时，该维度的问题会产生何种负面影响**。

### 评估结果摘要

| 维度       | 评分 | 等级      | 核心问题                                                            |
| ---------- | ---- | --------- | ------------------------------------------------------------------- |
| 衔接性     | 2/10 | 🔴 致命   | 因果链断裂（Step 2 网络异常→Step 3 凭空恢复）、连续4次返回无页面栈 |
| 逻辑清晰性 | 4/10 | ⚠️ 严重 | 二义性（"购物车或结算页"）、状态与布局矛盾（空白页+筛选按钮）       |
| 内容重复性 | 1/10 | 🔴 致命   | 44% 冗余（连续4次返回、2次加购、2次提交、2次相同导航）              |
| 顺序合理性 | 3/10 | 🔴 致命   | 异常无涟漪效应、54→1件数量矛盾、流程未完成（缺支付）               |
| 数据一致性 | 2/10 | 🔴 致命   | Step 14/15 "54件" vs Step 16 "去结算(1)" — 数量爆炸且无解释 |

**综合评分: 2.4/10 — 不可用于 AI Agent 仿真 APP 生成**

### 补充问题：原始 UTG 的步骤数量膨胀

原始 `utg_info.json` 中的 stepData 经常因为以下原因产生冗余步骤：

- **截图卡顿**：用户在同一页面停留时间较长，连续触发多次截图 → 同页面产生 2-4 个不同 `stepId` 的步骤
- **表述差异**：每张截图的 `ui_summary` 由 VLM 独立生成，措辞有细微差异（如 "展示35件商品…" vs "共有35件商品…"），文本字面匹配无法识别为重复
- **结果**：原始可能有 25-35 步，经过现有去重后仍有 18-20 步，远超合理范围（购物流程应 ≤10 步）

## 设计约束：Schema 与模板强制一致 🔴 硬性

> 所有优化和改动必须遵守此约束，违反即不合格。

### 约束来源

| 文件 | 角色 |
|------|------|
| `schema/model-schema.json` | **字段权威定义** — 所有字段名、类型、required、enum 以此为准 |
| `example_data/shopping-flow-search-and-buy_new.json` | **格式参考模板** — 输出的字段集合和 JSON 结构必须与此一致 |

### 关键约束

**1. FlowStep 字段锁死**

`model-schema.json` 中 `FlowStep` 定义：

```json
"FlowStep": {
  "required": ["order", "action"],
  "properties": {
    "order": { "type": "integer", "minimum": 1 },
    "action": { "type": "string" }
  }
}
```

- **仅有 `order` 和 `action` 两个字段**
- ❌ 禁止输出 `targetPage`、`boundMockId`、`anomalyType`、`screenKey` 等任何额外字段
- Phase 2 的 `FlowConverter._filter_step_fields()` 已实现模板字段过滤，确保输出不超出模板范围

**2. 所有字段定义以 schema 为准**

- 字段名必须与 `model-schema.json` 中的 `properties` key 完全一致（大小写敏感）
- `TopicField.type` 只能取 `["string", "number", "boolean", "array", "enum", "date", "price"]` 之一
- `Event.triggerType` 只能取 `["click", "scroll", "input", "gesture", "system", "timer"]` 之一
- `FlowStep.order` 必须 ≥1 且连续

**3. 不能凭空构造新字段**

- ❌ 不能在输出 JSON 的任何层级添加 schema 未定义的字段
- ❌ 不能修改 schema 定义的字段语义（如把 `action` 改成数组）
- ✅ 模板中存在但 schema 未定义的字段 → 保留模板原值，不做 LLM 生成

**4. 模板骨架保留**

- `events`、`topics[].fields`、`ui`、`baselineMapping` 等结构字段直接继承模板，不从零生成
- **动态生成字段**（LLM 从 step 描述中提取，当前处理无问题）：
  - `mainFlow.steps` — Phase 2 LLM 从 ui_summary 改写
  - `topics[].mockInstances` — Phase 2 LLM 从 steps 提取业务实体
- `topics[].mockInstances` 中字段值为 `null` 是合法的（数据源无此信息时不捏造）

### Phase 3 已有保护

`quality_validator.py` 的 `_validate_schema()` 已实现：
- 模板字段驱动验证（仅检查模板中存在的字段，避免对 schema 允许但模板未用的字段误报）
- `_get_template_step_fields()` 从模板 steps[0] 提取字段白名单
- `_filter_step_fields()` 按白名单过滤输出步骤

**所有优化方案在实现时必须遵守以上约束，不得破坏 Template ↔ Schema 的双向一致性。**

## 优化方案

### A — 步骤去冗增强 🔴 P0

#### A1. 文本指纹 + 语义兜底双层去重

**问题重述**：当前 `_compute_page_fingerprint()` 只比较 ui_summary 前 50 字符去标点后的字面值。卡顿导致的同页面多次截图，VLM 每次生成的 ui_summary 措辞略有不同，前 50 字不匹配 → 全部漏过。

**方案：纯文本语义去重（不读图片文件）**

```
L1 — 文本指纹快速初筛 (零成本，规则)
  │  比较 ui_summary 前 50 字去标点后是否完全相同
  │  ├─ 相同 → 直接判定同页面，合并
  │  └─ 不同 → 进入 L2
  │
L2 — LLM 语义等价判断 (极低成本，≤10 token/次)
  │  对连续两步的 ui_summary 全文，LLM 判断是否描述同一页面同一次浏览
  │  ├─ 相同语义 → 合并
  │  └─ 不同语义 → 保留各自
```

L2 Prompt 模板：

```
下面两个 UI 描述是否描述的是同一个页面、同一次浏览行为（仅是表述方式不同）？

A: "页面展示搜索结果列表，共有35件商品，包含华为畅享70X（1648元）"
B: "当前为搜索结果页，展示35件华为手机，用户正在浏览列表"

仅回答: same 或 different
```

- **每次判断 2-5 个 token**，整个序列去重总成本 < 200 token
- 不依赖图片文件，不需要 imageId → 截图路径映射
- 兼容所有 UTG 输入（包括无截图文件的情况）

**涉及文件**：`utg_preprocessor.py` → `deduplicate()` 方法增强

#### A2. compress_steps 默认启用

**问题**：`run_pipeline.py` 的 `--compress-steps` 默认关闭，导致 Phase 2 不执行相邻同页面步骤合并。

**方案**：

- 将 `--compress-steps` 改为 `--no-compress-steps`（即默认启用合并）
- 增强合并 Prompt 中的"连续返回"模式检测

**涉及文件**：`run_pipeline.py`、`flow_converter.py`

---

### B — 衔接性修复 🔴 P0

#### B1. 因果链断裂检测与修复 (Phase 0 新增)

**问题**：Step 2 终点状态为"网络连接失败，内容区域空白占位图"，Step 3 初始化状态为"搜索结果页展示筛选按钮"。中间缺少"网络恢复→页面数据加载"的过渡。

**方案**：在 Phase 0 预处理末尾（去重+重写完成后）增加因果链验证：

1. 对每对相邻步骤 (i, i+1)，检查 Step i 的"最终状态"是否能自然推导到 Step i+1 的"初始状态"
2. 若 Step i 描述中包含异常（加载失败/网络错误/空白/白屏），而 Step i+1 描述中无任何恢复/过渡语义 → 标记为断裂
3. LLM 修复：在断裂处插入或改写过渡描述

**涉及文件**：`utg_preprocessor.py` → 新增 `_validate_causal_chain()` + `_repair_causal_chain()`

#### B2. 异常涟漪传播增强 (Phase 1 增强)

**问题**：`enable_neighbor_adjust` 仅微调前后各 1 步。对于持续性异常（如网络失败），后续多步应持续体现异常影响。

**方案**：

- 邻居调整范围从前后各 1 步扩展到前后各 2 步
- REWRITE_PROMPT 增加持续性异常约束：
  > 如果注入的异常是持续性异常（网络失败、加载失败等），后续步骤的状态描述应反映该异常的残余影响，直到有明确的恢复操作
  >

**涉及文件**：`utg_anomaly_injector.py`

---

### C — 逻辑清晰性提升 🟡 P1

#### C1. 二义性描述检测 (Phase 3 新增)

**问题**：Step 13 的"购物车或结算页"同时给出两个可能目标页面，Agent 无法确定。

**方案**：在 QualityValidator 新增 `_validate_clarity()` 维度：

- 正则检测二选一结构：`或`、`或者`、`A/B`
- 正则检测不确定性词汇：`可能`、`大概`、`不确定`、`也许是`
- 每个匹配项标记为问题

**涉及文件**：`quality_validator.py` → 新增方法

#### C2. 状态-布局矛盾检测 (Phase 2 新增)

**问题**：Step 3 初始状态是"内容区域为空白占位图"，但布局描述包含"右侧筛选按钮"，用户无法在空白失败页上操作。

**方案**：在 Phase 2 生成 steps 后增加一轮状态一致性校验：

- LLM 检查：如果 action 中包含异常状态（加载失败/空白/网络错误），则对应的布局描述不应包含可交互控件（按钮/输入框等），除非明确说明"异常已恢复"

**涉及文件**：`flow_converter.py`

---

### D — 内容重复性修复 🔴 P0

#### D1. 操作重复检测 (Phase 3 新增维度)

**问题**：连续 4 次返回、2 次加购、2 次提交、2 次点击购物车——当前无任何环节检测。

**方案**：QualityValidator 新增冗余检测维度：

| 检测类型        | 方法                                 | 阈值                   |
| --------------- | ------------------------------------ | ---------------------- |
| 连续同操作      | 提取相邻步骤的操作动词，比较是否相同 | ≥2 次连续相同 → 标记 |
| 操作-目标对重复 | 提取 (操作动词, 操作目标) 元组       | ≥2 次出现 → 标记     |
| 同页面过度停留  | 检查 action 中的页面描述             | ≥3 步连续同页 → 标记 |
| 返回操作链      | 检测连续"返回"/"退回"操作            | ≥2 次连续 → 建议合并 |

**涉及文件**：`quality_validator.py` → 新增 `_validate_redundancy()` 维度

#### D2. 冗余步骤自动合并 (Phase 2 新增 pass)

**问题**：compress_steps 只合并同页面操作，无法处理分散的冗余（如 Step 14 和 Step 15 都是"点击购物车图标"但不在同一页面上下文中）。

**方案**：在 Phase 2 LLM 生成 steps 后新增"步骤精简 pass"：

- LLM Prompt 给定所有 steps，要求检测并合并以下模式：
  - N 次连续返回 → "用户连续返回至 XX 页"
  - 重复加购 → 保留一次
  - 重复导航 → 保留一次
  - 重复提交 → 保留一次
- 输出精简后的 steps，order 重新编号

**涉及文件**：`flow_converter.py`

---

### E — 数据一致性加固 🔴 P0

#### E1. 数量一致性检查 (Phase 3 增强)

**问题**：Step 14/15 "54件商品" → Step 16 "去结算（1）"— 数量暴增暴跌且无任何解释。

**方案**：在 `_validate_consistency()` 中增加数量追踪：

- 正则提取各步骤中数量表述：`\d+\s*件`、`共\s*\d+\s*件`、`去结算\s*[（(]\s*\d+`
- 追踪核心商品数量变化链
- 若某步数量与前一步差异 > 1 且无"添加"/"删除"/"清空"等操作 → 标记不一致

**涉及文件**：`quality_validator.py`

#### E2. 数据对齐 Prompt 增强 (Phase 0)

**问题**：`DATA_ALIGN_PROMPT` 只检查商品名称、价格、数量，未覆盖规格（颜色/存储）一致性。

**方案**：在检查要点中增加"规格一致性"、"配送信息一致性"检查项。

**涉及文件**：`utg_preprocessor.py` → 更新 `DATA_ALIGN_PROMPT`

---

### F — 流程合理性优化 🟡 P1

#### F1. 页面堆栈一致性验证 (Phase 3 新增)

**问题**：连续 4 次返回后页面堆栈混乱，Agent 无法确定当前位置。

**方案**：维护简化的页面堆栈模型：

- 从 action 中提取页面变化语义（"进入"/"跳转至" → push；"返回"/"退回到" → pop）
- 验证 pop 次数 ≤ push 次数
- 验证最后一次 pop 后的堆栈非空
- 标记无法匹配的"返回"操作

**涉及文件**：`quality_validator.py` → 新增 `_validate_page_stack()`

#### F2. 流程完成度检查 (Phase 3 新增)

**问题**：流程在"提交订单"处中断，缺少支付/订单确认等完成状态。

**方案**：根据业务类型（从 template 或 query 推断）建立预期终点：

- 购物类：expected_end = `["payment", "orderDetail", "orderComplete"]`
- 若最后 3 步内不包含任一终点 → 标记"流程未完成"

**涉及文件**：`quality_validator.py` → 新增 `_validate_flow_completeness()`

#### F3. 注入点合理性约束 (Phase 1 增强)

**问题**：异常注入在 Step 2（网络失败），但 Step 3 立即恢复正常操作，异常效果归零。

**方案**：在 `DECISION_PROMPT` 中增加约束：

> 如果异常是持续性异常（网络/加载/白屏），注入点之后至少保留 3 个步骤来展示异常影响，不应选择接近末尾的步骤

**涉及文件**：`utg_anomaly_injector.py`
