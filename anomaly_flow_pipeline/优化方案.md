# anomaly_flow_pipeline 优化方案

## 一、现状诊断

### 1.1 当前数据流

```
原始输入 (utg_info.json, N步)
    │
    ▼
[异常注入] ── LLM决策注入步 → LLM改写该步ui_summary
    │
    ▼
修改后 utg_info.json (仅1步ui_summary被改写)
    │
    ▼
[Flow转换] ── 盲替换: ui_summary → template.mainFlow.steps[].action
    │
    ▼
最终 Flow JSON (template结构 + raw ui_summary文本)
```

### 1.2 根因分析

通过对 `utg_info.json` 和 `诊断优化报告.md` 的交叉分析，发现质量问题分为**三层**：

| 层级 | 问题 | 根因代码位置 | 严重度 |
|------|------|-------------|--------|
| **数据源层** | utg_info.json 的 ui_summary 本身是"页面状态快照"而非"动作驱动描述" | 数据来自云端执行轨迹，非本管道控制 | **致命** |
| **数据源层** | 18步中~10步是冗余/循环的（Step10/11/12几乎相同页面） | 同上 | **严重** |
| **数据源层** | 商品名(描述iPhone vs 步骤华为)、价格(¥1400.8 vs ¥1648)不一致 | 同上 | **中等** |
| **数据源层** | 缺少ProductDetail、OrderDetail等关键页面；包含Messages无关页面 | 同上 | **严重** |
| **注入层** | LLM只改写1步，其余17步的原生质量问题全部保留 | `utg_anomaly_injector.py` | **致命** |
| **转换层** | flow_converter 做盲替换，不验证数据一致性 | `flow_converter.py` | **严重** |
| **转换层** | 输出 Flow 不符合 Schema 要求（缺 targetPage，action 不是动作描述） | `flow_converter.py` | **致命** |

### 1.3 新旧模板差异

`shopping-flow-search-and-buy_new.json` 相比旧版，**唯一变化**是新增了 `topics[].fields[].mockInstances` 数组：

- 8 个 mock 商品实例（iPhone 16 Pro、华为 Pura 70 Pro、小米 15 Pro、三星 S25 Ultra 等）
- 每个实例包含完整属性：brand、price、model、storage、color、processor、rating、salesVolume
- **这提供了数据一致性的"单一真相源"**，但当前 pipeline 完全未利用

## 三、各 Phase 详细设计

### Phase 0: utg 预处理器 — `utg_preprocessor.py`（新增）

**目标**: 在异常注入之前，先把原始 utg_info.json 质量问题修复到可接受水平。

#### 3.1 去重合并（Rule-based）

```python
# 策略: 按 ui_summary 语义相似度聚类，合并同一页面的连续重复步骤
def deduplicate_steps(stepData):
    # 1. 用 ui_summary 的前 50 个字计算 "页面指纹"
    # 2. 连续相同指纹的步骤合并为 1 步，保留第一个的 ui_summary
    # 3. 结果: 18步 → ~8-10步
```

**预期效果**: 参考诊断报告，18步可压缩至约8个有效状态。

#### 3.2 动作驱动重写（LLM-based）

为每步的 ui_summary 添加**因果链**：用户动作 → 系统响应。

```
原始: "页面顶部有搜索框，右侧有搜索按钮。页面主体显示商品推荐..."
改写: "用户在搜索框输入'华为手机'，系统跳转到搜索结果页，展示匹配的商品推荐列表"
```

**LLM Prompt 设计**:

```
你是一个 App 操作步骤描述优化专家。将页面状态描述改写为"用户动作 → 系统响应"格式。

原始描述: {ui_summary}
用户意图: {thought}
操作类型: {action_type}

输出格式（仅改写后的文本）:
"用户{动作描述}，系统{响应描述}，当前页面展示{关键UI信息}"
```

#### 3.3 数据对齐

从 `query` 字段提取商品实体（品牌、型号、价格区间），校验每步的数据一致性：

```python
def align_product_data(stepData, query):
    # 1. 从 query 解析商品实体: "京东/华为手机/3000以内"
    #    → brand=华为, category=手机, max_price=3000
    # 2. 扫描所有 step 的 ui_summary，标记不一致引用
    # 3. 对不一致步骤，LLM修正为统一商品引用
```

**关键**: 此时先做基础对齐，Phase 2 再结合模板的 mockInstances 做精准数据绑定。

#### 3.4 补充关键页面

```python
def add_missing_transition(stepData):
    # 检查是否有 "searchResult → cart" 的跳转
    # 缺少 ProductDetail → 在中间插入补充步骤
    # 补充步骤描述由 LLM 根据上下文生成
```

### Phase 2: 智能 Flow 转换 — 重写 `flow_converter.py`

#### 2.1 步骤 → screenKey 映射

利用 `baselineMapping.screenKeys`（from template）和每步的 ui_summary 语义，为每步分配 `targetPage`：

```python
SCREEN_KEY_MAP = {
    "首页": "home",
    "搜索": "search",
    "搜索结果": "searchResult",
    "商品详情": "productDetail",  # 当前缺失
    "购物车": "cart",
    "结算": "checkout",
    "支付": "payment",
    "订单": "orderDetail",
}
```

输出格式符合 Schema：

```json
{
  "order": 1,
  "action": "用户在搜索框输入'华为手机'，系统展示搜索结果",
  "targetPage": "searchResult",
  "anomalyTag": "dialog"  // 如有异常注入，标记异常类型
}
```

#### 2.2 数据绑定 — 利用 mockInstances

`shopping-flow-search-and-buy_new.json` 新增了 `mockInstances`，Phase 2 需要：

1. 从 `query` 匹配最佳 mockInstance
2. 将匹配的 mockInstance 数据注入到每步描述中
3. 确保同一步中商品名称、价格、SKU 完全一致

```python
def bind_mock_data(steps, mock_instances, query):
    # 1. 从 query 提取商品意图（品牌、品类）
    # 2. 匹配最相似的 mockInstance
    # 3. 为每步生成带具体商品数据的 action 描述
    # 4. 保证所有步骤引用同一 mockInstance
```

#### 2.3 Step 补齐策略

对于 utg 中没有但模板应有的关键页面（如 ProductDetail、OrderDetail）：

- **Option A**: 保留模板原有 steps 中对应位置的 action 描述
- **Option B**: 用 LLM 根据上下文生成补充步骤
- **推荐**: Option A（以模板为锚点，保证完整性）

## 五、LLM 调用量分析

| 当前 | 优化后 |
|------|--------|
| 每次注入 2 次 LLM 调用（决策 + 改写） | 见下方 |

优化后 LLM 调用（典型场景，18步 → 9步 → 15步）：

| Phase | LLM 调用次数 | token 消耗估计 |
|-------|-------------|---------------|
| Phase 0 去重 | 0（纯规则） | 0 |
| Phase 0 动作重写 | N步 × 1（约9次） | ~9 × 500 tokens |
| Phase 0 数据对齐 | 1次（批量修正） | ~2000 tokens |
| Phase 1 异常注入 | 2次 × M个异常 | 2M × 1000 tokens |
| Phase 2 智能转换 | 0-1次（screenKey分配合并） | ~2000 tokens |
| Phase 3 验证 | 0（纯规则） | 0 |

相比当前，LLM 调用增加约 **5x**，但这是**质量最关键的投入**。

## 六、实施路线图

### Step 1: Phase 0 — utg 预处理器（最高优先级）
- 实现 `utg_preprocessor.py`
- 动作：去重合并 + 动作驱动重写
- **预期质量提升**: 步骤数减少 40-50%，描述质量从"状态快照"提升为"动作驱动"
- **影响面**: 独立模块，不改现有代码

### Step 2: 模板切换 + Phase 2 智能转换
- 默认模板改为 `shopping-flow-search-and-buy_new.json`
- 重写 `flow_converter.py` 支持 screenKey 映射 + mockInstances 数据绑定
- **预期质量提升**: 输出符合 Schema，数据完全一致

### Step 3: Phase 1 增强 — 多步注入
- `utg_anomaly_injector.py` 支持多异常场景
- **预期质量提升**: 一次运行产出包含多个异常的综合测试流

### Step 4: Phase 3+4 — 质量闭环
- `quality_validator.py` + 报告输出
- **预期质量提升**: 可量化验证，避免低质量输出

## 七、MVP 建议

如果资源有限，建议按以下顺序实施（**价值/工作量比排序**）：

1. **Phase 0 去重 + 动作重写** — 最高 ROI，独立模块，解决核心问题
2. **模板切换为 _new.json** — 零开发量，但需要更新 config/start.sh
3. **FlowConverter 添加 targetPage 映射** — 解决 Schema 合规性
4. **质量验证器** — 发现问题的能力，在修复之前先建立检测
5. **多步注入** — 锦上添花功能

## 八、与主项目的对接

`anomaly_flow_pipeline` 当前的 README 声明 "本包是独立的，与项目原有 `ui_semantic_patch/` 目录完全无关"。优化后应保持这种独立性：

- 所有新增模块仅依赖 `requests` + `python-dotenv` + LLM
- 输出格式与 `model-schema.json` 对齐，便于后续集成到主项目
- `run_pipeline.py` 作为统一入口，封装完整流程
