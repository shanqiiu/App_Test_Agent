# 技术难题业界与项目方案对照

说明：

- 本表基于当前项目实际代码与文档整理。
- 当前项目主线是“基于真实截图序列的异常注入/语义补丁”，不是“APP Spec 代码生成”主线。
- 因此凡涉及 `APP Spec -> 代码 -> APP -> 自动验证` 的问题，当前方案列可能为空或标注“暂无系统方案”。

| 技术难题                                                                                                              | 业界先进技术方法                                                                                       | 当前方案                                                                                                                                               | 备注                                                                           |
| --------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------ | ------------------------------------------------------------------------------ |
| 人工创建长程复杂任务的评估基准成本高、周期长、覆盖维度单一且极易饱和                                                  | 真实环境 benchmark + 参数化任务实例化 + 执行式评测；典型路线包括 AndroidWorld、OSWorld、VisualWebArena | 基于真实截图序列做异常样本扩增；通过 `query_anomaly_mapping.json`、`generate_mapping.py`、`SequenceRewriter`、多异常模式批量生成测试样本         | 当前更偏“异常数据构造”，不是完整 benchmark 平台                              |
| 缺乏对智能体在信息不完整情况下能否识别关键缺失并主动澄清的系统性评估框架                                              | 澄清问题 benchmark、结构化不确定性建模、主动追问策略                                                   |                                                                                                                                                        | 当前仓库没有澄清回合模拟器、澄清质量指标或专门评测框架                         |
| 使用LLM生成测试任务集不够真实、准确                                                                                   | 基于真实 App/真实环境做参数化任务生成，保留人工审核；弱化纯自由生成                                    | `mapping-auto-generation-plan.md` 采用“自动生成初稿 + confidence 分级 + 人工审核”路线；尽量依赖真实截图序列而非纯 LLM 虚构                         | 已有较明确工程方向                                                             |
| 弹窗生成文字元素过多（控件、定位、关键词等）时，概率出现乱码                                                          | 避免让文生图模型直接生成大量可读 UI 文本；改用 OCR 定位、矢量文本层、程序化字体绘制                    | 文字类异常优先走 `TextOverlayRenderer`、`modify_text`、`modify_text_ocr`、`modify_text_ai`；`text_overlay.py` 使用 OCR/PIL/TrueType 字体重绘 | 文字编辑类已较强，`dialog` 模式仍存在 AI 生成文本过多的风险                  |
| 大模型指令遵从出现幻觉，缺少有效手段检测出现的幻觉                                                                    | 自一致性检测、多路验证器、生成后 verifier、基于 judge 的结构化评估                                     | `QualityVerifier` 对原图与异常图做双图校验，输出 `anomaly_present`、`semantic_match`、`visual_quality`、`naturalness`，并支持阈值与重试      | 已有工程化 verifier，但不是细粒度 hallucination taxonomy                       |
| 异常模式与生成内容映射关系只能通过规则方式，导致生成异常内容一致性概率性低                                            | 分类器 + 检索/规则 + 约束校验的混合方案；用封闭分类替代开放式自由决策                                  | `PageClassifier` 做封闭分类，`RuleEngine` 做确定匹配，`content_requirements` 做内容门禁，`expected_anomaly_mode` 做模式对齐                    | 已优于纯规则，但整体仍以规则体系为核心                                         |
| 大模型对相近截图当成同一个内容，导致注入点决策不准确                                                                  | 多帧上下文、历史轨迹建模、视觉 grounding、可访问树/结构化中间表示                                      | `SequenceAnalyzer` 将 `prev_page_info` 和 `step_context` 注入分类器，并对全序列候选点做对比打分，而非单帧一次性决策                              | 已部分缓解该问题                                                               |
| 截图中广告、其他信息导致大模型对截图理解不准确，导致注入点决策错误                                                    | 先做 UI parsing/OCR/grounding，再让 VLM 做高层语义判断                                                 | 主线为 `OmniParser` 本地解析 + `omni_vlm_fusion` 语义分组 + `RuleEngine`/`content_requirements` 门禁 + `ComponentPositionResolver` 定位      | 这是当前项目最强的工程落点之一                                                 |
| APP Spec复杂度比较高时（需要做一些验证），导致APP生成失败                                                             | DSL/Schema-first、分阶段生成、compiler/verifier loop、结构化约束校验                                   |                                                                                                                                                        | 当前项目主线不做整 APP 生成，更多是绕开该问题                                  |
| 异常模式与APP Spec转化依赖大模型，存在转化不准确的问题（弹窗位置不对，部分内容遮挡等）                                | 结构化 spec、布局约束、渲染后校验、视觉回归                                                            | `meta.json` + `MetaLoader` + `ComponentPositionResolver` + fallback，在截图层直接定位和合成，不经过“Spec 转 App”链路                           | 当前方案是绕开 Spec 转化误差，而不是正面解决                                   |
| APP Spec准确性验证缺少技术方法                                                                                        | Schema 校验、约束验证、视觉回归、执行式 UI 测试                                                        |                                                                                                                                                        | 当前没有 Spec validator；仅有图像级 `QualityVerifier` 与局部 diff 检查能力   |
| 生成APP主要功能自动验证（大模型可能将无法生成的内容自动跳过）                                                         | 执行式自动验证、E2E UI 测试、reward/constraint 检查                                                    |                                                                                                                                                        | 当前仅验证“异常图生成质量”，不验证“自动生成 App 的主功能完备性”            |
| APP代码与mock 数据抽象分类依赖经验，容易出现频繁重新生成APP（类似场景，不同query都要重新生成APP），生成效率低、成本高 | 模板化、组件库复用、Schema 复用、参数化实例化                                                          | 在异常资产层已有 GT 模板库、`meta.json`、规则表、mapping 自动生成思路                                                                                | 对“异常构造资产复用”部分有帮助；对“App 代码 + mock 数据抽象”暂无系统方案   |
| 同一任务Agent多次执行轨迹动态变化，无法采用固定规则判定                                                               | 多次 rollout、执行式 success metric、轨迹日志、鲁棒统计                                                | 可生成动态干扰样本，如 `response_delay` 通过复制前一帧制造连续两帧相同；保存 `history` 与 `decision_log`                                         | 数据侧有支撑，评估侧暂无统一判定器                                             |
| 缺少基于用户体验的判定技术，比如任务执行过程中与人交互频次、交互时机等体验判定                                        | 交互成本、澄清次数、轨迹长度、等待时延等体验指标建模                                                   |                                                                                                                                                        | 当前仓库没有 UX judge 或体验评分模块                                           |
| 智能体任务完成度评估方法不足，比如同一个场景使用多少任务测评、同一个任务执行多少次等                                  | 多任务、多变体、多次运行、pass rate/robustness/置信区间统计协议                                        |                                                                                                                                                        | 当前能批量生成多异常样本，但没有内建统计协议来定义样本量、重复次数与完成度口径 |

- 组件定位：`ui_semantic_patch/app/utils/component_position_resolver.py`
