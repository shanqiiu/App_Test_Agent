# 技术难题业界与项目方案对照

本文档只保留与当前仓库主线相关的问题。项目当前主线是“基于真实截图和轨迹的异常 UI 注入”，不是“从 APP Spec 自动生成整套应用”。

## 1. 对照表

| 技术难题 | 常见业界路线 | 当前项目落点 | 结论 |
|---|---|---|---|
| 人工构造长程任务成本高 | 真实环境 benchmark、参数化任务、执行式评测 | 用真实截图序列和 `mapping` 批量生成异常样本 | 更偏数据构造，不是完整 benchmark 平台 |
| 纯 LLM 生成测试任务不够真实 | 真实环境回放、半自动模板化生成、人工审核 | `generate_mapping.py` 负责生成初稿，运行时仍依赖真实截图或真实 `utg_info.json` | 当前路线是“自动生成初稿 + 人工把关” |
| 注入点自由决策不稳定 | 封闭分类、结构化中间表示、规则约束 | `PageClassifier + RuleEngine + SequenceAnalyzer` | 已从开放式决策收敛到可控方案 |
| 截图语义易受广告和噪音干扰 | UI parsing + OCR + grounding 后再做高层判断 | `OmniParser + omni_vlm_fusion + content_requirements` | 这是当前仓库最成熟的工程落点之一 |
| 文字较多时文生图容易乱码 | OCR 定位、程序化绘制、局部编辑替代整图生成 | `text_overlay.py`、`modify_text`、`modify_text_ocr` | 当前已经基本转向“定位后程序化修改” |
| 生成后缺少自动验收 | verifier、judge、视觉回归 | `QualityVerifier` 对异常存在性、语义一致性、自然度打分 | 有工程化校验，但仍是图像级，不是任务级 |
| 轨迹多次执行会漂移，固定规则难适配 | rollout 统计、结构化轨迹、按轨迹实时决策 | 视觉链路按当前截图序列决策，UTG 链路按 `ui_summary` 批量打分 | 数据侧支持动态轨迹，评测侧仍较弱 |
| 响应延迟类异常难以通过单图表达 | 时序注入、重复帧、状态冻结 | `SequenceRewriter` 直接插入前一帧副本 | 这是一个明确落在序列层的异常，而非渲染层 |
| 体验类评估难建模 | 交互成本、澄清次数、等待时延等指标 | 仓库内暂无专门体验评估模块 | 当前无系统方案 |
| 任务完成度统计口径不足 | 多次运行、置信区间、pass rate 协议 | 仓库能生成样本，但没有内建评测统计协议 | 当前无系统方案 |

## 2. 当前项目真正解决了什么

项目目前最明确解决的是以下三件事：

1. 如何把真实业务截图变成结构化异常样本。
2. 如何在一段真实操作轨迹中选择更自然的注入点。
3. 如何把异常注入结果稳定地组装回可回放的截图序列。

项目目前没有真正解决的是：

1. Agent 完成度评测协议。
2. 澄清式交互 benchmark。
3. 整 App 自动生成、执行和功能验收。

## 3. 与当前代码直接对应的实现位置

- 视觉序列决策：`ui_semantic_patch/app/injection/sequence_analyzer.py`
- 页面分类：`ui_semantic_patch/app/injection/page_classifier.py`
- 规则匹配：`ui_semantic_patch/app/injection/rule_engine.py`
- UTG 文本决策：`ui_semantic_patch/app/injection/utg_loader.py`、`ui_semantic_patch/app/injection/utg_decision.py`
- 序列改写：`ui_semantic_patch/app/injection/sequence_rewriter.py`
- 质量验证：`ui_semantic_patch/app/injection/quality_verifier.py`
- 单图渲染入口：`ui_semantic_patch/scripts/run_pipeline.py`
- mapping 生成：`data/data_process/generate_mapping.py`

## 4. 结论

当前仓库定位应该理解为：

- 一个异常 UI 样本生成与序列注入工具链
- 一个面向 Agent 测试数据构造的工程底座

而不是：

- 完整的 Agent benchmark 平台
- 通用的移动端应用自动生成系统
