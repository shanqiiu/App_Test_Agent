# 指令关键词解析机制说明

## 概述

当前系统中，`modify_text` 和 `image_broken` 等模式的**目标区域定位**使用纯正则匹配从用户指令中提取关键词，然后用这些关键词在 Stage 1 OmniParser 的 OCR 文字中进行匹配，找到对应的 UI 组件区域。

整个过程**零 VLM 调用**，完全确定性。

---

## 关键词提取规则

### 1. 去噪音词

先将指令中的功能词和虚词替换为空格：

```python
noise_words = [
    '将', '改为', '修改', '替换', '模拟', '异常场景',
    '的', '了', '在', '把', '被', '让', '使', '到', '和'
]
```

### 2. 提取中文词

正则 `[\u4e00-\u9fff]{2,}` → 提取所有连续 2 个及以上中文字符。

### 3. 提取英文/数字

正则 `[a-zA-Z0-9]+` → 提取所有连续英文字母或数字。

### 4. 去重 + 过滤

- 去重：相同关键词只保留一个
- 过滤：长度 < 2 的丢弃

---

## 关键词匹配打分

OCR 文字与关键词的匹配得分：

| 情况 | 得分公式 | 说明 |
|------|---------|------|
| 关键词完全出现在 OCR 文字中 | `len(kw) / len(ocr_text) * 2` | 精确匹配，权重最高 |
| 不匹配 | 拆为 2-gram 子串，每个命中 +0.3 | 模糊匹配 |

score > 0 即视为匹配成功。

## 示例

### 示例 1：票务场景

```
输入: "将z112次车硬卧改为灰色无票"

去噪后: "z112次车 硬卧 灰色 无票"
关键词: ["z112", "次车", "硬卧", "灰色", "无票"]
                                    ↑ "次车" 是噪音词"次"与"车"拼接的误提取
```

**问题**：`"改为"` 被整体作为噪音词删除，但 `"次车"` 是意外拼接产物。不过不影响匹配——`"z112"`、`"硬卧"` 等核心关键词仍在。

### 示例 2：电商遮挡场景

```
输入: "在购买页面注入遮挡层，覆盖鞋码选择区域，使鞋码选项无法正常显示和点击"

去噪后: "购买页面 注入 遮挡层 覆盖 鞋码选择 区域 使 鞋码选项 无法 正常 显示 点击"
关键词: ["购买页面", "注入", "遮挡层", "覆盖", "鞋码选择", "区域", "鞋码选项", "正常显示", "点击"]
```

**问题**：`"遮挡层"`、`"覆盖"` 等空间定位词被保留（image_broken 模式下），但对 `modify_text` 模式下这些词可能误导匹配。

### 示例 3：按钮置灰场景

```
输入: "将国家补贴按钮置灰"

去噪后: "国家补贴 按钮 置灰"
关键词: ["国家补贴", "按钮", "置灰"]
```

**问题**：如果 OCR 只检测到"国家补贴"文字（按钮图标无 OCR 文字），匹配到的区域可能是商品标题中的"国家补贴"字样，而非筛选按钮。

---

## 已知局限

| 局限 | 场景 | 影响 |
|------|------|------|
| 无法区分 UI 组件类型 | "国家补贴"可能是按钮 label 也可能是商品标题 | 可能匹配到错误区域 |
| 噪音词删除可能误删关键信息 | "改为"中的"为"可能是指令的目标词 | 关键词缺失 |
| 纯文字匹配无上下文理解 | "z112次"不会知道同页的"z156次"是不同车次 | 多卡片场景误匹配 |
| 英文提取可能断裂 | "iPhone 14 Pro" → ["iPhone", "14", "Pro"] | 复合词被拆散 |

---

## 与 VLM 方案的关系

关键词匹配**仅用于定位**，不用于语义编辑：

```
关键词匹配 → 定位目标区域（确定性）
     ↓
VLM 裁剪区分析 → 语义编辑内容（智能但慢）
     ↓
EditOp 执行
```

这样设计的意图：用快速的确定性方法找到"在哪"，用 VLM 解决"怎么改"。
