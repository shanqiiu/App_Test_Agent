# 异常注入与弹窗优化说明

## 一、异常注入决策优化

### 1.1 背景：旧方案的局限性

旧方案采用**固定中间位置注入**策略，即在截图序列的中间位置（`len(screenshots)//2`）注入异常，**无语义，无差异**。

```python
# 旧策略（batch_injection.py）
injection_point = len(screenshots) // 2
```

问题：
- 所有 App、所有场景的注入点相同（统一中间位置），无视页面结构
- 无法针对不同 App 的关键页面（如登录页、支付页、结果页）做差异化注入
- 注入的异常类型无上下文关联，生成的弹窗与当前页面语义不匹配

### 1.2 优化方案

新方案引入了**规则引擎 + VLM 页面分类 + AnomalyMappingResolver** 三层决策体系。

#### 1.2.1 映射配置驱动（AnomalyMappingResolver）

每个 App 的每个异常场景都有独立的映射配置：

```json
// config/query_anomaly_mapping.json（简化）
{
  "mappings": [
    {
      "query": "进入(铁路12306)购买火车票...",
      "fault_mode": "弹窗广告遮挡查询按钮",
      "fault_mode_key": "mode_1",
      "injection_config": {
        "anomaly_mode": "dialog",
        "instruction": "在首页生成广告弹窗遮挡查询功能"
      }
    },
    {
      "query": "进入(铁路12306)购买火车票...",
      "fault_mode": "无票状态提示",
      "fault_mode_key": "mode_2",
      "injection_config": {
        "anomaly_mode": "dialog",
        "instruction": "模拟查询结果无票的系统提示弹窗"
      }
    }
  ]
}
```

每个 App（铁路12306、瑞幸咖啡、去哪儿旅行）各定义 2 种故障模式，覆盖不同异常场景。

#### 1.2.2 规则引擎（RuleEngine）决定注入点

不再使用固定中间位置，而是**根据截图序列的内容语义决定哪个步骤最适合注入**：

- 规则定义在 `app/injection/rules.json` 中
- `SequenceAnalyzer` 使用 VLM 分析每张截图的页面类型（首页、搜索页、支付页等）
- `RuleEngine` 根据页面类型匹配规则，选出语义合理的注入点

例如：登录页 → 适合注入"登录过期弹窗"；支付页 → 适合注入"支付失败弹窗"。

#### 1.2.3 序列改写（SequenceRewriter）

注入后，原始序列的后续步骤需要**保留**而不是截断：

- 复制全部原始截图到 `modified_sequence/`
- 异常图替换注入点后第一个位置的原图
- 异常图只取 `final_*.png`，排除 `dialog_only`、`vis_bbox` 等中间产物

```python
# 关键改进
for i in range(len(original_screenshots)):      # 复制全部，不止到 injection_point
    shutil.copy2(original_screenshots[i], dst)

# 替换注入点后的原图，而非追加
original_at_idx.unlink()                         # 移除原图
shutil.copy2(anomaly_img, dst)                   # 替换为异常图
```

#### 1.2.4 完整流程

```
batch_injection_with_mapping.py
  ├─ AnomalyMappingResolver  → 匹配 query 对应的异常配置
  ├─ PageClassifier + VLM   → 分析每张截图页面类型
  ├─ RuleEngine             → 决策语义合理的注入点
  ├─ SequenceRewriter       → 调用 run_pipeline 生成异常 + 改写序列
  └─ QualityVerifier（可选）  → VLM 验证异常图片质量
```

## 二、弹窗关闭按钮的绘制可控性

### 2.1 问题：文生图模型自主生成关闭按钮

z-image-turbo 等文生图模型在生成弹窗时，受训练数据影响，**强制在右上角绘制 X 关闭按钮**，且不受 Prompt 否定指令控制。

### 2.2 解决方案：三层防御

采用**正向 Prompt + Negative Prompt + 后处理填充**三层策略。

#### 2.2.1 第一层：正面 Prompt 描述（`_build_ai_prompt_from_meta`）

从"告诉模型不要画什么"改为"告诉模型卡片是什么样的连续形状"：

```python
# 旧：否定指令（模型不遵守）
"the top-right corner is completely empty (no icon, no circle, no X mark, no button)"

# 新：正面形状描述（模型对形状的记忆优于否定）
"The dialog card has a CONTINUOUS, unbroken rounded rectangular shape.
 The top edge of the card is a straight horizontal line from left to right -
 there is no gap, no cutout, no symbol, no circle, and no icon of any kind
 at the top-right corner or anywhere on the top edge."
```

关键：否定句（"不要画 X"）对文生图模型效果极差，因为模型训练数据中"弹窗 → 右上角 X"的关联太强。改用正面描述"上边缘是一条从左到右的连续直线"。

#### 2.2.2 第二层：Negative Prompt

在 Negative Prompt 中封锁所有关闭按钮相关 token：

```python
# 中英文全覆盖
"close button, close icon, close symbol, X button, X icon, X mark, X symbol,
circle close, circular button, exit button, dismiss button, corner icon,
floating button, overlay button, close badge,
关闭按钮, 关闭图标, X图标, 叉号按钮, 角落图标, 角标按钮, 浮动按钮"
```

#### 2.2.3 第三层：后处理填充（`_erase_ai_close_button`）

在 AI 生成图像后，自动检测并**用弹窗背景色填充**关闭按钮区域：

```
检测：
  在弹窗右上角 40×40 区域做 X 形状对角线扫描
  → 两条对角线填充率均 > 20% → 判定为 X 按钮

填充：
  采样周围像素的背景色（白色/灰色中位数）
  → 用该颜色覆盖 X 按钮像素，而非设为透明

  之前: pixels[x,y] = (0,0,0,0)     → 挖空为透明，破坏矩形结构
  现在: pixels[x,y] = fill_color    → 背景色填充，保持矩形完整
```

这层是最终兜底，即使模型仍然画了关闭按钮，也会被自动修复。

#### 2.2.4 关闭按钮的绘制由代码负责

所有异常弹窗的关闭按钮统一由 `PatchRenderer._render_dialog_meta_driven` 在最终合成阶段用 PIL 绘制：

```python
# patch.py - 关闭按钮绘制（纯 PIL 代码，不依赖 AI）
button_layer = Image.new('RGBA', result_img.size, (0,0,0,0))
draw = ImageDraw.Draw(button_layer)
draw.ellipse([...], fill=bg_color)           # 圆形背景
draw.line([...], fill=x_color, width=... )     # X 符号
result_img = Image.alpha_composite(result_img, button_layer)
```

- 位置由 `meta.json` 的 `close_button_position` 字段控制（top-right / bottom-center 等）
- 样式由 `close_button_style` 控制（gray_circle_x / white_circle_x）
- AI 模型完全不参与关闭按钮的绘制，从根源上避免了模型生成劣质关闭按钮

### 2.3 总结

| 层 | 位置 | 策略 | 效果 |
|---|---|---|---|
| Prompt 正面 | `_build_ai_prompt_from_meta` | 描述卡片"连续矩形"形状 | 减少模型画 X 的倾向 |
| Negative Prompt | `generate_image_dashscope` | 20+ 关闭按钮负面词 | 封锁模型输出路径 |
| 后处理填充 | `_erase_ai_close_button` | X 形检测 + 背景色填充 | 最终兜底，100% 清除 |
| PIL 代码绘制 | `PatchRenderer` | 合成阶段统一绘制关闭按钮 | 保证关闭按钮质量 |
